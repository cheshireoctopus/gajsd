Deployment

• Deployment:

Deployment encompasses all the processes involved in getting new software or hardware up and running properly in its environment, including installation, configuration, running, testing, and making necessary changes

• The process of moving your local application to a remote server

• Once deployed, code moves from development to production

Static Websites:

• Purely front-end application

• Consist of only HTML, CSS and JS files; also includes images, videos, JSON, etc.

• Files are static in that they are served once and never dynamically updated

Deployment

Dynamic web applications:

• Defined as a client-server software application where the client runs in the browser

• An application with both a frontend and a backend

• Leveraged Firebase to build one:

⇒ HTML, CSS and JS are considered the client (frontend)

GitHub Pages

GitHub Pages

• Great option for deploying a simple static website

• Available for free with your GitHub acount (limited to one URL)

http://your_username.github.com

• Public (everyone can see your code) - do not push API keys or other sensitive information here

GitHub Pages

To deploy with GitHub pages:

1) Create your github.io repository

2) Clone your newly created repo to your local environment

3) Create HTML, CSS, and JS files

4) Add, commit, and push your newly created files

5) Visit http://your_username.github.com to see deployed code

GitHub Pages

Note: to properly activate GitHub pages, you must name the repo using the following convention:

your_github_username.github.io

GitHub Pages

• Where are you pushing your code to?

• A Server:

A server is a computer program or a machine that waits for requests from other machines or software (clients) and responds to them. The purpose of a server is to share data or hardware and software resources among clients. This architecture is called the client- server model.

• Re: web apps, the main purpose of a server is to listen to requests coming from clients (i.e. browsers) and respond with the appropriate resources to fulfill the request

• This could be the server redirecting to a whole new page and/or just providing some more data to the user interface

GitHub Pages

• When we pushed our code up to GitHub for GitHub Pages to host we essentially told GitHub Pages that we'd like to utilize their server to deliver our resources (HTML, CSS and JS)

• More specifically, we told GitHub Pages that whenever someone comes to http://username.github.io serve all the code from username/username.github.io

Deployment

Firebase Deployment

• Firebase Hosting provides developers the infrastructure, features, and tooling tailored to deploying and managing their websites

• Gives your project a subdomain on the firebaseapp.com domain:

YOUR_FIREBASE_APP.firebaseapp.com

• Using the Firebase CLI, you can deploy files from local directories to Firebase

• Files are served over an SSL connection from the closest edge server on their global CDN

Firebase Deployment

Firebase Deployment Overview:

1) Install the Firebase Command Line Interface (CLI)

2) Use firebase init to begin your Firebase project

3) Create the proper project directory using public/

4) Test your app locally using firebase serve

5) Use firebase deploy to delpoy your app to Firebase's servers

Firebase Deployment

• firebase.init creates a firebase.json config file:


  {
    "hosting": {
      "public": "public",
      "ignore": [
        "firebase.json",
        "**/.*",
        "**/node_modules/**"
      ]
    }
  }

• Only the public attribute is required; tells Firebase which directory to upload to Firebase hosting

• The ignore attribute specifies files which Firebase should ignore (smiliar to .gitignore)
